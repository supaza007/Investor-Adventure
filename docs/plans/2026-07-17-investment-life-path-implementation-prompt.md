## บริบทโปรเจกต์

โปรเจกต์เดิมคือเกมการ์ด deck-builder แนว Slay the Spire สอนการลงทุน
(React + Vite + Tailwind v4 + Electron) โครงสร้างปัจจุบัน:
- src/game/reducer.js — logic เกมทั้งหมดแบบ useReducer (629 บรรทัด, ระบบต่อสู้)
- src/components/ — MapScreen, EventScreen, RestScreen, RewardScreen, EnemyPanel,
  PlayerPanel, CardConfirm, CharacterSelect ฯลฯ (ทั้งหมดออกแบบมาสำหรับระบบต่อสู้เดิม)
- src/data/cards.json, classes.json, enemies.json — ข้อมูลการ์ด/คลาส/ศัตรู

ดีไซน์ใหม่ "เปลี่ยนแกนเกมทั้งหมด" (ต่อสู้ → จัดพอร์ต) ตามที่ระบุในเอกสารข้อ 9:
โค้ดเกมเดิม (reducer.js, EnemyPanel, CardConfirm) นำมาใช้ซ้ำได้น้อยมาก ควรวางแผนเป็นการ
เขียนระบบเกมใหม่เกือบทั้งหมด ส่วนที่ใช้ซ้ำได้แน่ๆ คือ enemies.json และ classes.json
(ปรับ schema) และ asset ภาพ/เสียง

## งานที่ต้องทำ

1. อ่านโค้ดปัจจุบันทั้งหมด (reducer.js, components/, data/) เพื่อเข้าใจ pattern
   ที่ใช้อยู่ (state shape, การจัดการ React state, สไตล์ Tailwind ที่ใช้)
2. เสนอแผน migration/rewrite ที่ชัดเจนก่อนเริ่มเขียนโค้ด — แตกเป็นเฟสที่ทดสอบได้ทีละเฟส
   (เช่น เฟส 1: ปรับ schema ข้อมูล → เฟส 2: state machine ใหม่สำหรับ 6 บทชีวิต →
   เฟส 3: Encounter Engine 5 สเตจ → เฟส 4: UI ใหม่ → เฟส 5: 4 สไตล์นักลงทุน →
   เฟส 6: ระบบแฟร์เนส/สรุปผล)
3. ก่อนเขียนโค้ดจริง ให้ช่วยตัดสินใจรายละเอียดที่ยังค้างอยู่ในเอกสารข้อ 8
   ("สิ่งที่ยังไม่ได้ตัดสินใจ") ร่วมกับผู้ใช้ก่อน โดยเฉพาะ:
   - สูตรคำนวณ outcome band (สเตจ 3) และ tag overlap ให้เป็นตัวเลขที่ทดสอบ balance ได้จริง
   - กลไกจุดตัดสินใจพฤติกรรม (สเตจ 4) แบบละเอียด
   - กลไกพิเศษของ Scammer event
   - ทุนเริ่มต้น/พลังงานต่อบท

## ข้อควรระวัง

- นี่คือ full rewrite ของ game logic ไม่ใช่ patch เล็กๆ — อย่าพยายามยัดกลไกใหม่เข้าไปใน
  reducer.js เดิมแบบ patch-on-patch เพราะ state shape เดิม (HP, block, energy ต่อเทิร์น)
  ไม่ตรงกับ state shape ใหม่ (portfolio value เป็นคะแนนสะสม, ตำแหน่งถือครองข้ามบท,
  exposure tag overlap)
- อย่าลบไฟล์ข้อมูลเดิม (enemies.json, classes.json) จนกว่าจะยืนยัน schema ใหม่แล้ว —
  ปรับ/ต่อยอด ไม่ใช่เขียนใหม่จากศูนย์ เพราะเนื้อหา (ชื่อเหตุการณ์, lesson text) ยังใช้ได้
- กลุ่มเป้าหมายคือนักเรียน ม.ปลาย ไม่มีพื้นฐานการเงิน — UI/ข้อความต้องเรียบง่าย
  ไม่ใช้ศัพท์การเงินที่ไม่อธิบาย
- เวลาเล่นจบ 1 รอบต้องอยู่ที่ ~10-15 นาที เป็นข้อจำกัดด้าน scope ที่ต้องยึดตลอดการออกแบบ
  รายละเอียด (จำนวนสเตจ, ความยาวข้อความ, จำนวนตัวเลือกต่อจุดตัดสินใจ)

เริ่มจากอ่านโค้ดปัจจุบันและเอกสารดีไซน์ให้ครบก่อน แล้วสรุปแผน migration ให้ผู้ใช้ตรวจ
ก่อนเริ่มเขียนโค้ดจริง
